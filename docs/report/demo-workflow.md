# KMU SW Navigator 데모 시연 워크플로우

이 문서는 발표영상 데모 시연에서 무엇을 테스트하고 어떤 순서로 보여줄지 정리한 자료입니다. 현재 프로젝트는 이미 VPS에 배포되어 있으므로, 기본 시연은 배포 URL에서 진행하는 것을 기준으로 합니다. 로컬 실행은 발표 당일 네트워크 또는 서버 문제가 생겼을 때의 보조 경로로만 사용합니다.

배포 URL은 발표 전 최신 주소로 교체합니다. 현재 확인한 보조 URL은 `https://dsl-catalogue-december-lookup.trycloudflare.com/`입니다.

## 1. 데모 목표

데모의 핵심 목표는 사용자의 입력이 Python 백엔드에서 처리되고, 그 결과가 웹 화면에 의미 있는 출력으로 표시되는 흐름을 보여주는 것입니다.

발표에서는 다음 네 가지를 중심으로 설명합니다.

1. RAG와 Mini LLM Wiki를 활용한 근거 기반 상담
2. 사용자 프로필 기반 개인화 추천
3. 채팅 입력에서 일정 후보를 추출하는 흐름
4. 일정 보드에서 D-Day를 계산하고 관리하는 흐름

교수님이 Python 구현 부분을 질문할 경우에는 RAG 검색, 추천 점수 계산, 일정 파싱, D-Day 계산을 중심으로 답변합니다.

## 2. 시연 전 점검

VPS 배포 환경에서 시연하기 전에 아래 항목을 확인합니다.

1. 배포 URL 접속이 되는지 확인합니다. 터널 URL을 쓰는 경우 발표 직전에 최신 URL인지 다시 확인합니다.
2. 로그인 화면이 정상적으로 뜨는지 확인합니다.
3. 테스트 계정으로 로그인할 수 있는지 확인합니다.
4. 온보딩 또는 프로필 입력 화면이 정상 동작하는지 확인합니다.
5. AI 상담 화면에서 질문을 보낼 수 있는지 확인합니다.
6. 답변에 내부 근거 문서가 표시되는지 확인합니다.
7. 추천 보드에서 추천 결과가 나오는지 확인합니다.
8. 채팅에서 과제 마감 문장을 입력했을 때 일정 후보 카드가 뜨는지 확인합니다.
9. 일정 후보를 저장했을 때 일정 보드에 D-Day가 표시되는지 확인합니다.
10. 완료/삭제 같은 일정 상태 변경이 되는지 확인합니다.

가능하면 발표 녹화 전에 코드 검증 명령도 한 번 실행합니다.

```powershell
pnpm docs:check
pnpm submission:check
pnpm test:backend
pnpm lint:backend
pnpm build:frontend
```

이 명령은 화면 시연 자체가 아니라 제출 코드가 문서, 과제 조건, 백엔드 테스트, 린트, 프론트엔드 빌드 기준을 통과하는지 확인하기 위한 것입니다.

## 3. 데모 시연 순서

권장 시연 순서는 다음과 같습니다.

```txt
배포 URL 접속
-> 로그인
-> 온보딩 또는 프로필 확인
-> RAG 상담 질문 입력
-> 근거 문서 확인
-> 추천 보드 확인
-> 채팅에서 과제 마감 문장 입력
-> 일정 후보 카드 확인
-> 일정 보드에서 D-Day 확인
-> LLM 활용 기록 설명
```

이 순서로 진행하면 사용자 입력, 조건문, 반복문, 함수, 리스트/딕셔너리, 의미 있는 출력, 실행 가능한 Python 코드, LLM 활용 기록 조건을 자연스럽게 설명할 수 있습니다.

## 4. 상세 시연 스크립트

### 4.1 배포 URL 접속

먼저 VPS에 배포된 서비스 URL에 접속합니다.

화면에서 보여줄 내용은 로그인 화면입니다. 이때 다음과 같이 설명합니다.

> KMU SW Navigator는 국민대학교 소프트웨어융합대학 학생을 위한 개인화 RAG 웹앱입니다. 사용자는 로그인 후 학업, 진로, 프로젝트, 일정 관련 질문을 입력하고, 시스템은 Python 백엔드에서 처리한 결과를 화면에 보여줍니다.

### 4.2 로그인

테스트 계정으로 로그인합니다. 발표 중에는 비밀번호가 노출되지 않도록 미리 입력하거나, 녹화 편집에서 입력 부분을 짧게 처리합니다.

확인할 내용은 다음입니다.

1. 로그인 성공 후 메인 화면으로 이동합니다.
2. 사용자 세션이 유지됩니다.
3. 이후 API 요청이 로그인 사용자 기준으로 처리됩니다.

설명 포인트는 Supabase Auth와 사용자별 데이터 분리입니다.

### 4.3 온보딩 또는 프로필 확인

온보딩 화면이나 설정/프로필 화면에서 사용자 정보를 입력하거나 이미 입력된 값을 확인합니다.

입력 예시는 다음과 같습니다.

```txt
학부: 소프트웨어학부
학년: 1학년
관심 분야: AI, 백엔드, RAG
목표: 진로 탐색, 프로젝트 준비
코딩 수준: beginner 또는 intermediate
```

이 장면에서는 사용자의 입력이 추천과 상담의 개인화 기준으로 사용된다는 점을 설명합니다.

### 4.4 RAG 상담 질문 입력

AI 상담 화면으로 이동해 질문을 입력합니다.

추천 질문은 다음 중 하나입니다.

```txt
AI 트랙을 준비하려면 어떤 과목을 보면 좋나요?
```

또는 다음 질문을 사용할 수 있습니다.

```txt
소프트웨어학부 신입생이 먼저 알아야 할 학교 시스템은 무엇인가요?
```

확인할 내용은 다음입니다.

1. 사용자가 질문을 입력합니다.
2. assistant 답변이 생성됩니다.
3. 답변 아래 또는 옆에 내부 근거 문서가 표시됩니다.
4. 답변 내용이 학교 자료와 관련되어 있습니다.

설명 포인트는 RAG입니다. 질문을 LLM에 바로 전달하는 것이 아니라 Python 백엔드가 내부 문서를 먼저 검색하고, 검색 근거를 포함해 답변을 생성한다고 설명합니다.

### 4.5 근거 문서 확인

답변에 표시된 근거 문서 또는 source chip을 보여줍니다.

이 장면에서는 다음과 같이 설명합니다.

> 답변만 생성하는 챗봇이 아니라, 어떤 내부 문서를 바탕으로 답변했는지 함께 보여줍니다. 이때 Mini LLM Wiki와 raw 문서 chunk를 검색해 질문과 관련된 근거를 찾습니다.

코드 설명으로 연결할 경우에는 `backend/app/services/retrieval_service.py`의 `LocalDocumentRetriever.search()`를 언급합니다.

### 4.6 추천 보드 확인

추천 보드로 이동해 개인화 추천 결과를 확인합니다. 필요한 경우 추천 조건을 직접 수정합니다.

추천 입력 예시는 다음과 같습니다.

```txt
관심사: AI, 백엔드
목표: 프로젝트 경험, 취업 준비
코딩 수준: beginner
활동 성향: 프로젝트형
```

확인할 내용은 다음입니다.

1. 추천 트랙 또는 추천 활동이 표시됩니다.
2. 추천 점수가 표시됩니다.
3. 추천 이유와 다음 행동이 표시됩니다.
4. 사용자 입력에 따라 추천 결과가 달라질 수 있음을 보여줍니다.

설명 포인트는 Python 규칙 기반 추천입니다.

> 추천 결과는 LLM이 임의로 만든 문장이 아니라, Python 코드의 리스트/딕셔너리 추천 규칙과 조건문 점수 계산으로 만들어집니다.

코드 설명으로 연결할 경우에는 `backend/app/services/recommendation_service.py`의 `TRACK_RULES`, `ACTIVITY_RULES`, `recommend_track()`, `recommend_activity()`를 언급합니다.

### 4.7 채팅에서 일정 후보 생성

다시 AI 상담 화면으로 이동해 과제 마감 문장을 입력합니다.

녹화용으로 가장 안정적인 입력은 절대 날짜를 포함한 문장입니다.

```txt
데이터베이스 보고서 6월 11일까지 제출해야 해
```

상대 날짜 처리까지 보여주고 싶다면, 시연 전에 한 번 더 확인한 뒤 다음 문장을 사용합니다.

```txt
자료구조 과제 다음주 금요일까지 제출해야 해
```

확인할 내용은 다음입니다.

1. assistant 답변 아래 일정 후보 카드가 표시됩니다.
2. 제목, 과목, 마감일, D-Day가 표시됩니다.
3. 사용자가 확인 버튼을 눌러 내부 일정으로 저장할 수 있습니다.

설명 포인트는 자연어 입력 처리입니다.

> 사용자가 일반 문장으로 과제와 마감일을 입력하면, 백엔드가 일정 의도를 감지하고 날짜 표현을 분석합니다. LLM parser를 사용할 수 있지만 최종 D-Day 계산은 Python 날짜 함수로 다시 수행합니다.

코드 설명으로 연결할 경우에는 `backend/app/services/assignment_service.py`의 `preview_assignment_from_text()`, `calculate_d_day()`, `format_d_day()`를 언급합니다.

### 4.8 일정 보드에서 D-Day 확인

일정 보드로 이동해 방금 추가한 일정이 표시되는지 확인합니다.

확인할 내용은 다음입니다.

1. 일정 제목이 표시됩니다.
2. 마감일이 표시됩니다.
3. D-Day가 표시됩니다.
4. 완료 버튼을 눌렀을 때 상태가 변경됩니다.
5. 삭제 버튼으로 일정을 삭제할 수 있습니다.

이 장면은 과제 조건 중 의미 있는 출력과 사용자 입력 기반 처리 결과를 가장 명확하게 보여줍니다.

### 4.9 LLM 활용 기록 설명

마지막으로 LLM 활용 기록을 짧게 설명합니다. 현재 앱 화면에는 별도 LLM 기록 탭을 두지 않았으므로, 보고서의 LLM 활용 기록 표 또는 `docs/llm/usage-log.md`를 보여줍니다.

설명할 내용은 다음입니다.

1. Codex와 ChatGPT는 기능 설계, 구현 보조, 오류 분석, 보고서 정리에 사용했습니다.
2. Gemini API는 RAG 답변 생성과 일정 파싱에 사용했습니다.
3. 주요 Python 로직은 함수, 조건문, 반복문, 테스트 기준으로 직접 설명할 수 있게 정리했습니다.
4. 추천, RAG 검색, 일정 파싱, D-Day 계산은 Python 코드로 분리하고 테스트했습니다.

## 5. 데모 중 강조할 기술

발표에서 강조할 기술은 다음 순서가 좋습니다.

1. RAG
   - 내부 문서 검색
   - Mini LLM Wiki
   - 근거 기반 답변

2. Python 백엔드
   - FastAPI route
   - service 계층 분리
   - Pydantic schema
   - 사용자별 데이터 처리

3. 추천 시스템
   - 리스트/딕셔너리 기반 추천 후보
   - 조건문 점수 계산
   - 사용자 입력에 따른 결과 변화

4. 일정 처리
   - 자연어 일정 입력
   - 날짜 표현 분석
   - D-Day 계산
   - 일정 저장, 완료, 삭제

5. LLM 활용 기록
   - LLM 사용 목적 기록
   - 직접 검토/수정 내용 기록
   - 테스트와 빌드로 검증

## 7. 데모 실패 대비

VPS 환경에서 시연할 때는 네트워크, 서버 상태, 외부 API 키 문제가 생길 수 있습니다. 발표 전에는 아래 백업을 준비합니다.

1. 정상 동작 화면을 미리 녹화합니다.
2. RAG 답변, 추천 결과, 일정 후보, D-Day 화면을 스크린샷으로 저장합니다.
3. 테스트 계정 정보를 미리 확인합니다.
4. 배포 URL 접속이 안 되면 로컬 실행 또는 녹화본으로 대체합니다.
5. Gemini 또는 외부 API가 느릴 경우, 일정 파싱은 Python fallback 경로를 설명하고, RAG 답변 생성은 미리 녹화한 결과 화면이나 스크린샷으로 대체합니다.

로컬 보조 실행이 필요할 때만 다음 명령을 사용합니다.

```powershell
pnpm dev:backend
pnpm dev:frontend
```

데모 마지막에는 다음과 같이 정리합니다.

> 지금까지 로그인, 온보딩, RAG 상담, 추천 보드, 채팅 기반 일정 후보, 일정 보드 D-Day 흐름을 시연했습니다. 이 프로젝트는 사용자의 입력을 Python 백엔드에서 처리하고, 조건문, 반복문, 함수, 리스트/딕셔너리 기반 로직을 통해 답변, 추천, 일정 결과를 출력합니다. LLM은 개발 보조와 답변 생성에 활용했지만, 핵심 판단 로직은 직접 설명 가능한 Python 코드로 분리하고 테스트했습니다.
