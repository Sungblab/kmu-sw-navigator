# KMU SW Navigator 최종보고서

## 1. 프로젝트 개요

본 프로젝트의 이름은 **KMU SW Navigator**입니다. 주제는 국민대학교 소프트웨어융합대학 학생을 위한 **개인화 RAG 기반 AI 학업 내비게이터**입니다. 사용자는 웹 화면에서 학업, 트랙, 진로, 동아리, 과제 일정에 대해 질문하거나 자신의 관심사와 목표를 입력할 수 있습니다. Python FastAPI 백엔드는 이 입력을 바탕으로 학교 자료 검색, 개인화 추천, 채팅 기반 일정 후보 생성 결과를 제공하도록 구현했습니다.

프로젝트의 핵심 개발 언어는 Python입니다. React 프론트엔드는 사용자가 입력하고 결과를 확인하기 위한 화면 역할을 담당합니다. 실제 과제 조건을 만족하는 입력 처리, 조건문, 반복문, 함수, 리스트와 딕셔너리 기반 판단 로직은 FastAPI 백엔드의 Python 코드에 구현했습니다.

이 프로젝트는 단순한 챗봇이 아니라, 국민대학교 소프트웨어융합대학 학생에게 필요한 정보를 내부 문서와 Mini LLM Wiki에서 검색하고, 그 근거를 바탕으로 답변과 추천을 제공하는 웹앱입니다. 신입생 안내를 주요 시연 시나리오로 유지하되, 구조상 소프트웨어융합대학 전체 학년의 학업, 진로, 프로젝트, 일정 관리를 지원하는 방향으로 설계했습니다.

## 2. 프로젝트 주제와 목적

국민대학교 소프트웨어융합대학 학생은 수강신청, 전공 트랙, 졸업요건, 진로 준비, 동아리 활동, 과제 일정처럼 여러 종류의 정보를 계속 확인해야 합니다. 그러나 필요한 정보는 학교 홈페이지, 학부 공지, 교과과정 문서, 동아리 자료, 개인 일정에 흩어져 있어 처음 학교 시스템을 접하는 학생이 한 번에 이해하기 어렵습니다.

KMU SW Navigator의 목적은 이러한 정보 탐색 부담을 줄이는 것입니다. 사용자가 자연어로 질문하면, 시스템은 내부에 정리된 학교 자료를 먼저 검색하고, 검색 결과와 사용자 프로필을 함께 사용해 답변합니다. 또한 사용자의 학년, 관심 분야, 목표, 코딩 수준을 바탕으로 트랙과 활동을 추천하고, 채팅 문장에 포함된 과제 마감 정보를 일정 후보로 제안한 뒤 사용자가 확인하면 저장합니다.

프로젝트의 구체적인 목적은 다음과 같습니다.

1. 학교와 학부 자료를 기반으로 답변하는 RAG 챗봇을 구현했습니다.
2. 사용자의 관심사와 목표를 반영하는 개인화 추천 기능을 제공했습니다.
3. 채팅에 포함된 과제와 마감 문장을 Python 코드로 분석해 일정 후보와 D-Day 결과를 출력했습니다.
4. LLM을 개발 보조와 답변 생성에 활용하되, 핵심 판단 로직은 직접 설명 가능한 Python 코드로 구성했습니다.
5. LLM 사용 목적, 결과, 직접 검토 내용을 기록해 제출 조건을 만족하도록 했습니다.

## 3. 주요 기능

### 3.1 RAG 기반 AI 상담

사용자는 채팅 화면에서 학업, 트랙, 진로, 일정 관련 질문을 입력합니다. 백엔드는 질문을 받은 뒤 내부 문서 검색기를 호출해 관련 문서 조각을 찾고, 검색된 근거와 개인화 메모리를 함께 사용해 답변을 생성합니다.

예를 들어 사용자가 "AI 트랙을 준비하려면 어떤 과목을 봐야 하나요?"라고 질문하면, 시스템은 교과과정, 트랙, 로드맵 관련 문서를 검색하고, 관련 근거를 답변과 함께 반환합니다. 이 방식은 LLM이 아무 근거 없이 답을 만드는 것을 줄이고, 프로젝트가 수집한 학교 자료를 중심으로 답변하도록 만들었습니다.

### 3.2 Mini LLM Wiki

Mini LLM Wiki는 RAG 검색에 사용되는 내부 지식 기반입니다. `data/wiki` 폴더에는 커리큘럼, 트랙, 진로, 동아리, 학교 시스템, 신입생 안내 등 학생 상담에 필요한 자료를 주제별 Markdown 문서로 정리했습니다.

원천 자료는 `data/raw`에 저장하고, 이를 상담에 적합한 형태로 요약한 문서를 `data/wiki`에 두었습니다. 백엔드는 이 문서를 chunk 단위로 나누어 검색할 수 있게 만들고, 검색 결과를 `evidence.internal_sources` 형태로 API 응답에 포함합니다.

### 3.3 개인화 추천

추천 기능은 학생의 학년, 관심사, 목표, 코딩 수준, 학습 선호를 입력으로 사용합니다. Python 코드에서는 추천 후보를 리스트와 딕셔너리로 정의하고, 조건문을 통해 점수를 더합니다. 최종적으로 점수가 높은 트랙이나 활동을 먼저 보여줍니다.

이 기능은 LLM이 임의로 추천을 만드는 방식이 아니라, 발표에서 설명 가능한 규칙 기반 Python 로직으로 구현했습니다. 예를 들어 3학년 이상이고 AI, 백엔드, RAG에 관심이 있으면 포트폴리오형 AI 백엔드/RAG 프로젝트 트랙의 점수가 올라가도록 했습니다.

### 3.4 채팅 기반 일정 후보와 D-Day 계산

일정 기능은 별도의 추출 입력창이 아니라 AI 상담 흐름과 연결했습니다. 사용자가 채팅에서 "자료구조 과제 다음주 금요일까지야"처럼 과제와 마감일이 포함된 문장을 보내면, 백엔드는 기존 일정 parser로 제목, 과목, 마감일, D-Day를 계산하고 `assignment_preview` action을 응답에 포함합니다.

프론트엔드는 assistant 답변 아래에 "일정 후보" 카드를 보여주고, 사용자가 "캘린더에 추가" 버튼을 누른 경우에만 기존 내부 일정 저장 API로 저장합니다. 바로 자동 저장하지 않고 확인 버튼을 거치게 한 이유는 채팅 문장을 잘못 해석했을 때 사용자가 저장 전 확인할 수 있게 하기 위해서입니다. 저장된 일정은 일정 보드에서 D-Day, 완료, 삭제 상태로 관리합니다.

### 3.5 LLM 활용 기록

프로젝트는 LLM 사용이 허용되는 과제 조건에 맞게 사용 목적과 결과를 기록합니다. 개발 보조 과정은 `docs/llm/usage-log.md`에 남겼고, 앱 내부 기능에서도 채팅 답변 생성, Google grounding, 일정 파싱, 문서 임베딩 같은 LLM 사용을 기록할 수 있도록 `llm_usage_logs` 구조를 두었습니다.

## 4. 시스템 구조

전체 구조는 프론트엔드, 백엔드, 데이터 저장소, RAG 문서 시스템으로 나눌 수 있습니다.

| 영역 | 사용 기술 | 역할 |
| --- | --- | --- |
| Frontend | React, Vite, TypeScript | 사용자가 로그인하고 질문, 추천 입력, 일정 후보 저장을 수행하는 화면 |
| Backend | Python, FastAPI, Pydantic | API 요청 처리, RAG 검색, 추천, 일정 후보 계산, LLM 기록 |
| Database | Supabase Auth, Postgres, pgvector | 사용자별 프로필, 메모리, 채팅, 일정, LLM 로그 저장 |
| RAG 자료 | `data/raw`, `data/wiki`, Mini LLM Wiki | 학교/학부 자료를 구조화하고 검색 근거로 제공 |
| LLM | Gemini API, Google Search grounding | 답변 생성, 일정 파싱, 최신 정보 보강 |

사용자 입력이 들어온 뒤의 흐름은 다음과 같습니다.

1. 사용자가 React 화면에서 질문이나 추천 조건을 입력합니다.
2. 프론트엔드는 `frontend/src/lib/api.ts`의 API 함수로 FastAPI 백엔드에 요청을 보냅니다.
3. 백엔드는 인증된 사용자 ID를 기준으로 프로필, 메모리, 채팅 기록을 조회합니다.
4. 질문이 채팅이면 RAG 검색기로 내부 문서 근거를 찾습니다.
5. 필요한 경우 Gemini API에 검색 근거와 사용자 정보를 포함한 프롬프트를 전달합니다.
6. 백엔드는 답변, 추천, 일정 후보, D-Day, 근거, LLM 사용 기록을 구조화된 응답으로 반환합니다.
7. 프론트엔드는 응답을 화면에 표시하고, 사용자는 근거 문서, 추천 결과, 채팅에서 제안된 일정 후보를 확인합니다.

## 6. 주요 코드 설명

### 6.1 채팅 API 처리 흐름

주요 파일은 `backend/app/api/chat.py`입니다. 이 파일의 `_build_and_store_chat_response()` 함수는 채팅 질문 처리의 중심입니다.

이 함수는 먼저 사용자 메모리를 조회하고, `retriever.search(request.message)`를 호출해 질문과 관련된 내부 문서를 검색합니다. 이후 `build_chat_response()`를 통해 답변 구조를 만들고, 필요한 경우 LLM 사용 기록도 저장합니다. 마지막으로 채팅 저장소에 사용자 질문과 assistant 답변을 저장합니다.

이 부분은 사용자 입력이 실제 Python 함수에 전달되고, 검색과 답변 생성, 저장이라는 의미 있는 처리 결과로 이어지는 흐름을 보여줍니다.

### 6.2 RAG 검색 로직

주요 파일은 `backend/app/services/retrieval_service.py`입니다. `LocalDocumentRetriever.search()` 함수는 사용자의 질문을 토큰화하고, 내부 문서 payload를 반복문으로 순회하면서 점수를 계산합니다.

검색 로직은 다음 기준을 사용합니다.

- 질문에서 한글, 영어, 숫자 토큰을 추출합니다.
- 각 문서의 제목, category, heading path, content를 검색 대상 문자열로 합칩니다.
- 질문 토큰이 포함되면 점수를 부여합니다.
- 점수가 같으면 `wiki_page`를 `raw_document`보다 우선합니다.
- 점수가 0보다 큰 문서만 결과에 포함합니다.

이 코드는 반복문, 조건문, 함수, 리스트를 모두 사용하며, 질문 입력에 따라 다른 검색 결과를 반환합니다. 따라서 과제 조건을 설명하기 좋은 핵심 Python 코드입니다.

### 6.3 LLM 답변 프롬프트 구성

주요 파일은 `backend/app/services/answer_generation_service.py`입니다. `build_answer_prompt()` 함수는 사용자 질문, intent, 개인화 메모리, 내부 검색 근거, 첨부 자료를 하나의 프롬프트로 구성합니다.

이 함수에서 중요한 점은 LLM에게 단순히 질문만 전달하지 않는다는 것입니다. Python 코드가 먼저 근거와 사용자 정보를 정리하고, LLM은 그 구조화된 입력을 바탕으로 자연어 답변을 만듭니다. 검색 근거가 없을 때는 "내부 자료 근거 없음"을 명시해 LLM이 없는 출처를 지어내지 않도록 했습니다.

### 6.4 개인화 추천 로직

주요 파일은 `backend/app/services/recommendation_service.py`입니다. `TRACK_RULES`와 `ACTIVITY_RULES`는 추천 후보를 담은 리스트/딕셔너리 구조이고, `recommend_track()` 함수는 입력값에 따라 각 후보의 점수를 계산합니다.

예를 들어 관심사 키워드가 후보의 keyword와 일치하면 점수를 더하고, 3학년 이상이면 포트폴리오형 프로젝트 추천 점수를 더합니다. 코딩 수준이 beginner이면 기초 웹 프로젝트 트랙의 점수를 높입니다. 마지막에는 점수순으로 정렬해 가장 적합한 추천을 반환합니다.

이 로직은 LLM이 추천을 임의로 생성한 것이 아니라, Python 조건문과 점수 규칙으로 결과를 만든다는 점에서 중요합니다.

### 6.5 채팅 일정 후보와 D-Day 계산

주요 파일은 `backend/app/services/assignment_service.py`와 `backend/app/api/chat.py`입니다. `preview_assignment_from_text()` 함수는 자연어 과제 문장을 일정 미리보기로 변환하고, 채팅 API는 일정 의도 문장에 대해 `assignment_preview` action을 assistant 응답에 붙입니다.

처리 흐름은 다음과 같습니다.

1. 사용자가 채팅에서 과제와 마감일이 포함된 문장을 입력합니다.
2. 백엔드는 일정 의도를 감지하고 기존 일정 parser를 호출합니다.
3. Gemini parser가 있으면 먼저 자연어 문장을 JSON 형태로 파싱합니다.
4. parser가 실패하거나 없으면 Python 규칙 기반 parser를 사용합니다.
5. "오늘", "내일", "다음주 금요일", "6월 11일" 같은 날짜 표현을 조건문과 정규식으로 분석합니다.
6. 마감일을 기준으로 D-Day를 계산합니다.
7. 제목, 과목, 마감일, D-Day, 신뢰도, 누락 필드를 `assignment_preview` action으로 반환합니다.
8. 프론트엔드는 답변 아래 일정 후보 카드를 보여주고, 사용자가 확인 버튼을 누르면 내부 일정 저장 API로 저장합니다.

D-Day 계산은 LLM 결과를 그대로 믿지 않고 `calculate_d_day()`와 `format_d_day()` 같은 Python 함수로 수행합니다. 이 부분은 사용자의 채팅 입력을 의미 있는 일정 후보와 저장 가능한 일정 결과로 바꾸는 기능입니다.

### 6.6 프론트엔드 API 연결

프론트엔드의 주요 API 연결 파일은 `frontend/src/lib/api.ts`입니다. `streamChatMessage()` 함수는 채팅 메시지를 `/api/chat/stream`으로 보내고, 서버가 보내는 SSE 이벤트를 읽어 화면에 답변을 실시간으로 표시합니다.

프론트엔드는 과제의 핵심 언어는 아니지만, 사용자가 Python 백엔드에 입력을 전달하고 결과를 확인하는 실제 실행 화면을 제공합니다. 발표에서는 프론트엔드를 제품 시연용 화면으로 설명하고, 핵심 처리 로직은 Python 백엔드 중심으로 설명하는 것이 적절합니다.

## 7. LLM 활용 방식

이 프로젝트에서는 ChatGPT, Codex, Gemini 등 LLM을 사용했지만, LLM이 생성한 코드를 그대로 제출하지 않기 위해 사용 목적과 결과를 기록하고, 주요 코드는 직접 검토했습니다.

LLM은 다음 목적으로 활용했습니다.

1. 프로젝트 주제와 기능 범위 정리
2. PRD, spec, plan, 보고서, 발표 흐름 작성 보조
3. FastAPI, RAG, Supabase, Gemini 연동 코드 초안 작성 보조
4. 테스트 실패 원인 분석과 수정 방향 제안
5. 코드 리뷰와 문서-코드 불일치 점검
6. 발표에서 설명할 핵심 Python 로직 정리

최종 구현에서는 LLM 결과를 그대로 복사하지 않고, 다음 방식으로 직접 검토했습니다.

- 추천 로직은 Python 리스트/딕셔너리와 조건문 점수 계산으로 정리했습니다.
- 일정 파싱은 채팅 기반 일정 후보 생성에 사용했고, LLM parser가 있어도 Python fallback과 D-Day 재계산을 유지했습니다.
- RAG 검색 결과는 `RetrievalResult` 구조로 통일해 근거를 확인할 수 있게 했습니다.
- FastAPI API 응답은 Pydantic schema로 고정했습니다.
- `pnpm test:backend`, `pnpm lint:backend`, `pnpm build:frontend`, `pnpm docs:check`, `pnpm submission:check` 같은 명령으로 동작과 제출 조건을 검증했습니다.

### 7.1 LLM 사용 기록 요약

아래 표는 제출 보고서 안에서 바로 확인할 수 있도록 정리한 LLM 활용 기록 요약입니다. 전체 개발 기록은 별도 문서에 남겼지만, 이 보고서만 제출하더라도 어떤 목적으로 LLM을 사용했고 어떤 부분을 직접 검토했는지 확인할 수 있도록 핵심 항목을 포함했습니다.

| 날짜 | 사용 도구 | 사용 목적 | 결과와 직접 검토 내용 |
| --- | --- | --- | --- |
| 2026-05-13 | Codex | 프로젝트 주제와 기능 범위 정리 | 국민대 소프트웨어융합대학 학생을 위한 개인화 RAG 웹앱으로 주제를 확정하고, 신입생 지원에서 전체 학년 지원 방향으로 범위를 확장했습니다. |
| 2026-05-14 | Codex | RAG 문서 ingest와 검색 구조 설계 | raw/wiki Markdown을 chunk로 나누고, `RetrievalResult` 구조로 검색 근거를 반환하도록 구현했습니다. 문서 검색 우선순위와 근거 부족 처리 기준은 직접 검토했습니다. |

LLM을 사용한 결과물은 그대로 제출하지 않았습니다. 예를 들어 추천 기능은 LLM이 만든 문장을 그대로 사용하는 방식이 아니라, `TRACK_RULES`, `ACTIVITY_RULES` 같은 리스트/딕셔너리와 조건문 점수 계산으로 다시 정리했습니다. 일정 기능도 LLM parser가 있더라도 최종 D-Day 계산은 Python 날짜 함수로 수행했습니다. RAG 답변 생성 역시 검색 근거, 사용자 메모리, 첨부 자료를 Python 코드에서 먼저 구조화한 뒤 LLM에 전달하도록 구현했습니다.

또한 개발 과정에서 작성한 코드는 테스트와 빌드로 확인했습니다. 백엔드 서비스 테스트, FastAPI API 테스트, 문서 검사, 제출 조건 검사, 프론트엔드 빌드를 실행해 LLM이 제안한 코드가 실제로 동작하는지 검증했습니다. 오류가 발생한 경우에는 원인을 확인한 뒤 직접 수정했습니다. 예를 들어 배포 서버에서 일정 저장 API가 500 오류를 냈을 때는 Supabase에 `datetime` 객체가 JSON 직렬화되지 않은 상태로 전달된 것이 원인임을 확인했고, Supabase 저장소 테스트를 추가한 뒤 payload를 JSON 직렬화 가능한 값으로 바꾸었습니다.

## 8. 실행 결과

실행 결과는 발표자료와 발표영상에서 실제 화면 캡처와 시연 흐름으로 보여줄 예정입니다. 보고서에는 다음 결과를 포함할 수 있습니다.

### 8.1 로그인 및 온보딩

사용자는 Supabase Auth 기반 로그인 화면에서 계정을 사용해 접속합니다. 첫 사용자는 학부, 학년, 교과과정, 관심 분야, 목표, 코딩 수준, 학습 선호를 입력합니다. 이 정보는 사용자별 프로필과 개인화 메모리로 저장되어 이후 상담과 추천에 사용됩니다.

### 8.2 RAG 상담 결과

사용자가 학업이나 트랙 관련 질문을 입력하면 채팅 화면에 답변이 표시됩니다. 답변에는 내부 자료 근거가 함께 표시되어 어떤 문서를 바탕으로 답변했는지 확인할 수 있습니다.

예상 시연 질문은 다음과 같습니다.

- "AI 트랙을 준비하려면 어떤 과목을 보면 좋나요?"
- "소프트웨어학부 신입생이 먼저 알아야 할 학교 시스템은 무엇인가요?"
- "백엔드와 RAG에 관심이 있는데 어떤 활동을 해보면 좋나요?"

### 8.3 추천 결과

추천 보드에서는 사용자의 관심사, 목표, 코딩 경험을 바탕으로 트랙과 활동 추천을 확인할 수 있습니다. 결과에는 추천 제목, 점수, 추천 이유, 다음 행동이 포함됩니다.

### 8.4 채팅 일정 후보 및 D-Day 결과

채팅 화면에서는 과제와 마감일이 포함된 문장을 입력해 일정 후보를 만들 수 있습니다. 예를 들어 "자료구조 과제 다음주 금요일까지야"라고 입력하면 assistant 답변 아래에 일정 후보 카드가 표시됩니다. 사용자가 "캘린더에 추가" 버튼을 누르면 내부 일정으로 저장되고, 일정 보드에서 D-Day, 완료, 삭제 상태를 확인할 수 있습니다.

### 8.5 검증 결과

제출 전 기본 검증으로 다음 명령을 사용할 수 있습니다.

```powershell
pnpm docs:check
pnpm wiki:build
pnpm submission:check
pnpm test:backend
pnpm lint:backend
pnpm build:frontend
```

기존 제출 체크리스트에는 `pnpm verify:local`, `pnpm live:readiness`, `pnpm live:smoke-run` 등 더 넓은 검증 명령도 정리되어 있습니다. 최종 제출 전에는 실제 실행 결과를 다시 확인하고, 발표자료와 보고서에 최신 검증 결과를 반영할 예정입니다.

보고서 작성 시점 기준으로 확인한 주요 검증 결과는 다음과 같습니다. `pnpm docs:check`는 필수 문서 20개를 확인했고, `pnpm submission:check`는 사용자 입력, 조건문, 반복문, 함수, 리스트/딕셔너리, 의미 있는 출력, 실행 가능한 Python 코드, LLM 활용 기록 등 제출 조건 증거를 확인했습니다. 일정 저장 오류 수정 이후에는 Supabase 저장소 테스트와 assignment API 테스트를 실행해 일정 저장 payload가 JSON 직렬화 가능한 값으로 전달되는지 확인했습니다.

실제 시연에서 보여줄 화면은 로그인/온보딩 화면, AI 상담 화면, 근거 문서 표시, 추천 보드, 채팅 답변 아래 일정 후보 카드, 일정 보드의 D-Day 목록입니다. 이 흐름은 사용자가 입력한 문장이 Python 백엔드에서 처리되고, 그 결과가 다시 웹 화면에 의미 있는 출력으로 표시되는 전체 과정을 보여줍니다.

## 9. 한계점 및 개선 방향

첫 번째 한계는 자료 최신성입니다. RAG 시스템은 내부 문서와 Mini LLM Wiki를 기반으로 동작하므로, 학교 공지나 교과과정이 변경되면 문서를 다시 정리하고 ingest해야 합니다. 이를 개선하려면 공식 페이지 자료를 주기적으로 확인하고 변경 사항을 자동으로 감지하는 기능이 필요합니다.

두 번째 한계는 추천 로직의 정교함입니다. 현재 추천은 발표에서 직접 설명 가능한 규칙 기반 점수 계산으로 구현했습니다. 이 방식은 과제 조건을 만족하고 동작을 이해하기 쉽지만, 실제 서비스로 발전시키려면 더 많은 학생 피드백과 데이터 기반 평가가 필요합니다.

세 번째 한계는 외부 서비스 의존성입니다. Supabase, Gemini, Google grounding 같은 외부 서비스는 실제 SaaS형 구조를 구현하는 데 유용하지만, API 키나 네트워크 상태에 따라 시연이 영향을 받을 수 있습니다. 그래서 로컬 fallback과 검증 명령을 함께 준비했습니다.

향후 개선 방향은 다음과 같습니다.

1. 더 많은 공식 학교 자료를 수집해 Mini LLM Wiki를 확장합니다.
2. RAG 검색 품질 평가를 자동화하고, 검색 결과의 정확도를 개선합니다.
3. 추천 결과에 사용자 피드백을 반영해 개인화 정확도를 높입니다.
4. Google Calendar 또는 eCampus와 같은 외부 일정 시스템 연동을 고도화합니다.
5. 실제 배포 환경에서 사용자별 데이터 격리와 보안 검증을 강화합니다.

## 10. 결론

KMU SW Navigator는 국민대학교 소프트웨어융합대학 학생을 위한 Python 중심 개인화 RAG 웹앱입니다. 사용자는 질문, 추천 조건, 과제 마감 문장 같은 입력을 제공하고, 시스템은 이를 바탕으로 학교 자료 기반 답변, 개인화 추천, 채팅 기반 일정 후보와 D-Day 계산 결과를 출력합니다.

이 프로젝트는 과제에서 요구한 사용자 입력, 조건문, 반복문, 함수, 리스트/딕셔너리, 의미 있는 출력, 실행 가능한 Python 코드, LLM 활용 기록 조건을 충족합니다. 또한 LLM을 개발과 답변 생성에 활용하되, 핵심 판단 로직은 Python 코드로 분리하고 테스트와 문서로 검증했습니다.

최종 발표에서는 RAG와 Mini LLM Wiki를 핵심 기술로 소개하고, Python 백엔드의 채팅 API, RAG 검색, 추천 점수 계산, 채팅 일정 후보와 D-Day 계산 코드를 중심으로 설명할 예정입니다. 시연에서는 로그인, 온보딩, RAG 상담, 추천 결과, 채팅에서 일정 후보를 추가하는 흐름과 일정 보드의 D-Day 결과를 보여주면 프로젝트의 목적과 구현 결과를 가장 명확하게 전달할 수 있습니다.
